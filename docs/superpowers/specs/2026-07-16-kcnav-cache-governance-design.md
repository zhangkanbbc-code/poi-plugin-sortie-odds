# KCNav 缓存治理设计（v0.4.0 / v0.4.1）

日期：2026-07-16
状态：已与用户确认设计，待实施

## 背景与目标

插件 v0.3.4 依赖 KCNav（tsunkit.net）实时获取地图拓扑、敌编成与带路核验数据。用户曾收到
`401 automation` 拒绝，担心被封。本次改造目标：**大幅降低对 KCNav 的实时依赖，使请求行为
接近人类使用，被拒后能自动降级且状态可见。**

现状中的高危行为：

1. `runAnalysis` 用 `Promise.allSettled` 并发拉取所有战斗点敌编成（一次 4~8 个并发请求）。
2. 带路核验 `fetchRouteEntries` 在每次编成/路线变化时触发（仅 450ms 防抖），且无磁盘缓存。
3. 收到 401 后无冷却机制，下一次状态变化立即重试。
4. 活动图（世界 ≥10）不写磁盘，每次重启 poi 全部重拉。

## 方案总览

- **v0.4.0（方案 B）运行时请求管治**：串行队列 + 401 熔断 + 全图磁盘缓存 + 核验收敛 + UI 状态。
- **v0.4.1（方案 A）内置通常图快照**：开发侧脚本一次性抓取全部通常图数据打包进插件，
  日常通常图零请求。

两个里程碑独立发版：v0.4.0 先发布试用几天，再发 v0.4.1。

## 第 ① 部分：请求管治层（v0.4.0）

新增 `src/services/kcnav-transport.ts`，`kcnav.ts` 中所有网络请求改经该模块。

### 串行请求队列

- 全插件单一队列，任何时刻最多 1 个在途 KCNav 请求。
- 相邻请求强制间隔 1.5~3 秒（均匀随机抖动），自上一请求**完成**时刻起算。
- 两个优先级：`interactive`（用户点击"分析"/"刷新本图数据"/"试探一次"）排在
  `background`（自动跟随、预热、带路核验）之前；同优先级 FIFO。
- 入队前先查缓存；缓存命中的请求不进队列。现有内存缓存（30 分钟）与 in-flight 去重保留。

### 401 熔断器

- 收到 automation 拒绝（HTTP 401 且响应体含 `automation`）→ 进入冷却，时长 6 小时起步。
- 冷却状态持久化到 `sortie-odds-cache/kcnav-state.json`（字段：`cooldownUntil`、
  `cooldownLevel`），poi 重启不重置。
- 冷却期间所有请求立即以 `KcnavCooldownError` 拒绝，不发网络包；上层照常回退磁盘缓存/
  内置快照。
- UI 提供"试探一次"：放行单个探测请求（interactive 优先级）。成功 → 解除熔断并清零等级；
  再次 401 → 冷却翻倍（6h → 12h → 24h，24h 封顶）。
- 超时、5xx 等其他错误不触发熔断，按现有逻辑回退陈旧缓存；队列间隔即自然退避。

### 带路核验收敛

- 核验结果磁盘缓存：`evidence-<hash>.json`，键为「mapId + 路线边 ID 序列 + 编成特征
  （RoutingFleetFeatures 全字段）」的稳定哈希。
- TTL：通常图 30 天，活动图 24 小时。
- 触发防抖 450ms → 3 秒；仅在插件面板可见时触发（`document.visibilityState` 或组件挂载状态）。
- 同一「图+路线+特征」组合命中缓存时不发请求。

## 第 ② 部分：缓存层级（v0.4.0）与内置快照（v0.4.1）

### 统一磁盘缓存 maps-v2

- 缓存根：`sortie-odds-cache/maps-v2/<mapId>/`，通常图与活动图都写盘。
- 文件结构与 v1 相同：`map.json`、`enemy-<edgeId>.json`、`evidence-<hash>.json`，
  外层包 `{ savedAt, value }`。
- TTL：
  - 通常图（世界 1~9）：地图/敌编成 30 天（与现状一致）。
  - 活动图（世界 ≥10）：地图 24 小时、敌编成 12 小时。
- v1 → v2 迁移：v2 未命中时回读 `normal-maps-v1` 同名文件并写入 v2，不发网络请求。
- 网络失败回退陈旧缓存的现有行为保留，活动图同样适用。

### 内置通常图快照

- 开发侧脚本 `scripts/build-map-snapshot.ts`（node/tsx 运行，不进构建产物）：
  - 按 3 秒+抖动节流，串行抓取全部通常图（1-1～1-6 及 2-1～7-5，共约 36 图）的
    `map.json` 与全部战斗边 `enemycomps`。
  - 输出到源码 `data/normal-maps/<mapId>/`，结构与磁盘缓存一致；另写
    `data/normal-maps/manifest.json`（生成日期、图列表）。
  - 全量运行约 20 分钟（36 图 × ~9 请求 × 3s）。
- 打包：`package.json` 的 `files` 增加 `data/`；运行时用 `join(__dirname, 'data', ...)`
  以 fs 读取（与 `engine/` 同模式），预计体积 2~3MB。
- UI 显示快照日期（来自 manifest）。

### 数据解析顺序

通常图：

1. 磁盘缓存（未过期）
2. **内置快照**
3. 磁盘陈旧缓存
4. 网络（走队列）

→ 快照覆盖的通常图永远不发请求；仅游戏新增而快照缺失的图才走网络。
"刷新本图数据"强制走网络并写盘；TTL 内磁盘新数据优先于快照。

活动图：

1. 磁盘缓存（未过期）
2. 网络（走队列）
3. 磁盘陈旧缓存

## 第 ③ 部分：UI 变化（v0.4.0，快照日期显示属 v0.4.1）

- KCNav 状态 Tag 三态：
  - `正常`（绿）
  - `冷却中 · 剩余 Xh`（红，旁附"试探一次"按钮）
  - `离线 · 使用本地数据`（灰，网络错误但非熔断时）
- 数据来源与年龄：路线行小字显示，如"敌编成数据：内置快照（2026-07-16）"或
  "本地缓存 · 3 天前"；多节点来源混合时按最旧者显示。
- 控制行新增"刷新本图数据"按钮：强制重拉当前图地图+当前路线敌编成（串行过队列），
  冷却期间禁用。
- 缺数据报错文案更新：提示可点"刷新本图数据"或等待冷却结束。

## 测试（vitest，沿用 tests/ 现有风格）

- 队列：假计时器验证串行执行、间隔下限、interactive 插队。
- 熔断：401 → 状态文件写入 → 冷却期请求即拒 → 到期解除 → 再次 401 翻倍、24h 封顶。
- 解析顺序：通常图与活动图各分支逐一断言（磁盘新/快照/磁盘旧/网络的优先关系）。
- 核验缓存：同一「图+路线+特征」第二次不发请求；任一特征变化则发。
- v1 → v2 迁移读取。

## 追加：出击自动跟随增强（v0.4.3，2026-07-16 用户确认）

- 海域跟随：每次出击开始（`live.startedAt` 变化）无条件同步海域输入为当前出击图；
  手动改图仍允许，下次出击自动拉回。
- 目标点全自动选取：地图加载或实际路线推进时，自动选取从当前实际路线**可达**的第一个
  boss 点（选项已 boss 优先排序）；无可达 boss 则选第一个可达战斗点。手动选择只在当前
  节点内生效，下次进点重算（用户选择"完全自动"模式）。
- 实现：纯函数 `pickAutoTarget(map, actualEdges)` 进 `src/services/route.ts` 并配单测；
  视图 effect 以值键（edges join 字符串）触发。

## 追加：战斗机制可视化与特攻阵形（v0.6.0，2026-07-17 用户确认）

背景：弹观/对空CI/先制反潜/特攻在 vendor 模拟器 MECHANICS 中默认全部启用（mechanics:{}
为合并语义），一直在参与胜率；缺口是 (a) 特攻对阵形有要求而 adapter 阵形写死，特攻从不触发；
(b) 用户看不到模拟考虑了什么。

- **inspect 接口**：runner.js 新增 `inspect` 消息——`SIM.createSimFleet(fleetF, 0, false)`
  构建舰队后逐舰返回 `{ oasw: canOASW(), aaciTypes: AACItype, dayCutin: !!canAS(),
  specialAttack: canSpecialAttackUnique(ship,false,true) }`；并对阵形候选逐一重建舰队、
  用 `canSpecialAttackUnique(ship,false,false)` 实测哪些阵形能触发特攻（specialFormations）。
  判定与胜率模拟同源。
- **能力标注 UI**：逐舰显示 先制反潜/对空CI种别/弹观(需制空)/特攻(所需阵形)。
- **目标点阵形选择**：下拉「自动/单纵/复纵/轮形/梯形/单横」（联合舰队为第一~第四警戒）；
  自动=特攻可用时选其所需阵形，否则沿用现有启发式。adapter 增加 targetFormation 覆盖
  最后一个节点的 formationOverride。
- 道中节点阵形仍用启发式；每节点完整编辑仍在 roadmap。

## 风险与备注

- 快照数据随游戏版本可能过时（通常图敌编成罕见变动）：依赖插件发版更新；
  用户也可用"刷新本图数据"以磁盘缓存覆盖。
- 抓取脚本本身是一次性开发侧行为，用慢节流执行以避免触发 automation 判定。
- 项目不使用 git（用户已确认）；版本快照依靠里程碑发布的 tgz 包留存。
